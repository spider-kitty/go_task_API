# Go Task API

![Go](https://img.shields.io/badge/Go-Backend-00ADD8?logo=go&logoColor=white)
![PostgreSQL](https://img.shields.io/badge/PostgreSQL-Database-4169E1?logo=postgresql&logoColor=white)

A production-style RESTful Task Management API built with **Go**, **PostgreSQL**, and **Docker Compose**.

The project demonstrates layered backend architecture, dependency inversion through repository interfaces, persistent data storage, validation, structured error handling, automated database migration, and unit testing.

---

## Highlights

- Complete task CRUD operations
- PostgreSQL persistence
- Dockerized API and database
- Layered architecture: Handler, Service, Repository
- Repository abstraction with multiple storage implementations
- Filtering and text search
- Input validation and structured errors
- Automatic database migration
- Unit tests for handlers, services, and validation
- Secure configuration through environment variables
- Persistent Docker volume for database data

## Architecture

```text
HTTP Client
     |
     v
Handler Layer
     |
     v
Service Layer
     |
     v
Repository Interface
     |
     +-------------------+-------------------+
     |                   |                   |
     v                   v                   v
PostgreSQL           JSON Storage       Memory Storage
```

### Layer Responsibilities

| Layer | Responsibility |
|---|---|
| Handler | Receives HTTP requests and returns HTTP responses |
| Service | Contains business logic and validation flow |
| Repository | Defines storage operations through an interface |
| PostgreSQL Repository | Persists task data in PostgreSQL |
| Migration | Creates the required database table automatically |

This design keeps the HTTP and business layers independent from the selected storage technology.

## Getting Started with Docker

### Prerequisites

Make sure the following tools are installed:

- Git
- Docker Desktop
- Docker Compose

### 1. Clone the Repository

```bash
git clone https://github.com/YOUR_GITHUB_USERNAME/go-task-api.git
cd go-task-api
```

Replace `YOUR_GITHUB_USERNAME` with your actual GitHub username.

### 2. Create the Environment File

On Linux or macOS:

```bash
cp .env.example .env
```

On Windows PowerShell:

```powershell
Copy-Item .env.example .env
```

Open the newly created `.env` file and configure the required values:

```env
POSTGRES_USER=taskuser
POSTGRES_PASSWORD=change_this_password
POSTGRES_DB=taskdb
APP_PORT=8080
```

> The `.env` file contains local credentials and must never be committed to Git.

### 3. Validate the Docker Compose Configuration

```bash
docker compose config -q
```

If the command produces no output, the Compose configuration is valid.

### 4. Build and Run the Project

Run the API and PostgreSQL with visible logs:

```bash
docker compose up --build
```

Run the services in the background:

```bash
docker compose up --build -d
```

### 5. Check Container Status

```bash
docker compose ps
```

Expected service status:

```text
db    Up (healthy)
api   Up
```

### 6. Stop the Project

```bash
docker compose down
```

The PostgreSQL data remains available because it is stored in a named Docker volume.

To remove the containers and database data:

```bash
docker compose down -v
```

> Warning: The `-v` option permanently removes the PostgreSQL Docker volume and all stored database records.

## Environment Variables

| Variable | Description | Example |
|---|---|---|
| `POSTGRES_USER` | PostgreSQL username | `taskuser` |
| `POSTGRES_PASSWORD` | PostgreSQL password | `change_this_password` |
| `POSTGRES_DB` | PostgreSQL database name | `taskdb` |
| `APP_PORT` | Host port used by the API | `8080` |

The API receives its database connection string through the `DATABASE_URL` environment variable generated by Docker Compose.

The connection string follows this structure:

```text
postgres://username:password@db:5432/database_name?sslmode=disable
```

Inside the Docker network, the API connects to PostgreSQL using the Compose service name `db` instead of `localhost`.

## API Endpoints

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/health` | Check API availability |
| `POST` | `/tasks` | Create a new task |
| `GET` | `/tasks` | Get all tasks |
| `GET` | `/tasks/{id}` | Get a task by ID |
| `PUT` | `/tasks/{id}` | Update an existing task |
| `DELETE` | `/tasks/{id}` | Delete a task |

## Filtering and Search

The task list endpoint supports filtering and text search through query parameters.

| Request | Description |
|---|---|
| `/tasks?status=todo` | Filter tasks by status |
| `/tasks?category=backend` | Filter tasks by category |
| `/tasks?search=golang` | Search through task content |
| `/tasks?status=todo&category=backend` | Combine multiple filters |

Example:

```http
GET /tasks?status=todo&category=backend
```

## Create a Task

Request:

```http

```

Request body:

```json
{
  "title": "Dockerize the API",
  "description": "Run the Go API and PostgreSQL with Docker Compose",
  "status": "todo",
  "category": "backend"
}
```

### cURL Example

```bash
curl -X POST http://localhost:8080/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Dockerize the API",
    "description": "Run the Go API and PostgreSQL with Docker Compose",
    "status": "todo",
    "category": "backend"
  }'
```

### Windows PowerShell Example

```powershell
$body = @{
    title       = "Dockerize the API"
    description = "Run the Go API and PostgreSQL with Docker Compose"
    status      = "todo"
    category    = "backend"
} | ConvertTo-Json

Invoke-RestMethod `
    -Method Post `
    -Uri "http://localhost:8080/tasks" `
    -ContentType "application/json" `
    -Body $body
```

Example response:

```json
{
  "id": 1,
  "title": "Dockerize the API",
  "description": "Run the Go API and PostgreSQL with Docker Compose",
  "status": "todo",
  "category": "backend"
}
```

## Docker Architecture

The application runs as two separate Docker services:

```text
Client
  |
  | HTTP request
  v
Go API Container
  |
  | PostgreSQL connection
  v
PostgreSQL Container
  |
  v
Persistent Docker Volume
```

The API and PostgreSQL communicate through the private network automatically created by Docker Compose.

The PostgreSQL database is not required to expose port `5432` to the host operating system because the API accesses it internally through:

```text
db:5432
```

## Docker Volume

PostgreSQL data is stored in a named volume:

```text

```

Stopping or recreating the containers does not remove stored task data.

This command preserves the database data:

This command removes the database data:


## Security Practices

The project follows several basic security practices:

- Database credentials are not hard-coded in Go source files.
- Local credentials are stored in `.env`.
- `.env` is excluded from Git through `.gitignore`.
- `.env` is excluded from the Docker build context through `.dockerignore`.
- `.env.example` documents required variables without exposing real credentials.
- The database connection string is supplied through environment variables.
- PostgreSQL is accessed through the private Docker Compose network.
- Real passwords and secrets are not included in the public repository.

The project supports multiple repository implementations:

```text
Repository Interface
├── PostgreSQL Repository
├── JSON Repository
└── Memory Repository
```

Because the service layer depends on the repository interface instead of a specific database implementation, the storage system can be changed without rewriting the business logic or HTTP handlers.
